# ChessTrainer AI Coach – Player Pattern Analysis

## Objective

Extend ChessTrainer to analyze multiple games from the same player and detect recurring strategic and tactical patterns.

Instead of analyzing a single game, the system will identify **long-term strengths and weaknesses**.

This mimics the behavior of a human chess coach.

---

# Concept

Single game analysis answers:

"What happened in this game?"

Pattern analysis answers:

"What does this player consistently do wrong?"

Example:

Recurring patterns may include:

- weak king safety
- frequent tactical oversights
- poor endgame technique
- slow development

---

# Data Source

The analysis uses data generated by the existing pipeline.

For each game:

PGN  
→ Stockfish analysis  
→ feature extraction  
→ ML predictions  
→ coaching report

Each game generates structured data.

---

# Stored Dataset

Each analyzed game should store a structured record.

Example schema:

game_id
player
opening
result
accuracy
critical_moments
tactical_tags
error_labels
training_topics


These records create a **training dataset of the player's behavior**.

---

# Pattern Detection

Patterns can be detected using simple aggregation.

Example metrics:

- frequency of blunders
- tactical motifs missed
- openings played
- error distribution by phase

Example:

fork errors: 23
pin errors: 11
endgame blunders: 5


---

# Pattern Categories

Patterns can be grouped into categories.

## Tactical Patterns

Examples:

- forks
- pins
- discovered attacks
- mating threats

---

## Strategic Patterns

Examples:

- weak pawn structure
- loss of central control
- bad piece coordination

---

## Opening Patterns

Examples:

- recurring mistakes in a specific opening
- poor development
- premature attacks

---

## Endgame Patterns

Examples:

- poor king activity
- lost winning endgames
- lack of technique

---

# Pattern Detection Algorithm

Steps:

1 collect all analyzed games  
2 extract features and error labels  
3 aggregate metrics  
4 detect high-frequency issues  
5 generate pattern report

# ChessTrainer AI Coach – Player Style Classification & ML-Based Coaching

**Last Updated:** March 18, 2026

## Objective

Extend the AI coaching system to incorporate **player style classification** using ML model predictions.

Instead of generic coaching advice, the system will:
- Classify player style using ML cluster models (XGBoost/RandomForest/Neural Networks)
- Generate personalized feedback based on detected playing style
- Cross-reference ML predictions with Stockfish analysis for validated insights

This document defines how to integrate ML-based style classification into the existing LLM coaching pipeline.

---

## System Context

**Technology Stack:**
- Backend: Python (FastAPI)
- LLM: Ollama (llama3.1:8b)
- ML Models: XGBoost, RandomForest, Neural Networks
- Engine: Stockfish
- Database: PostgreSQL

**Available Inputs:**
- `pgn`: Standard game notation
- `stockfish_metadata`: 
  - Move-by-move evaluation (centipawns)
  - Best move vs actual move
  - Average centipawn loss (ACPL)
  - Clock times
- `ml_features`: 
  - Extracted game features
  - Model predictions with confidence scores
  - **Player style clusters** (e.g., "aggressive", "speculator", "solid", "positional")

**Goal:** Use Ollama to orchestrate these data sources and generate a **narrative pedagogical report** that humans can understand, avoiding redundant tactical calculations already performed by Stockfish.

---

## Implementation Strategy

### Phase 1: ML Style Classification (PLANNED)

**Objective:** Train ML models to classify players into style clusters based on game features.

**Player Style Taxonomy:**
1. **Aggressive** - High risk, tactical play, frequent attacks
2. **Positional** - Strategic, long-term planning, solid structure
3. **Solid** - Defensive, low error rate, patient play
4. **Tactical** - Combination-focused, sharp positions
5. **Speculator** - High variance, risky moves, unbalanced positions

**Data Source:**
- Historical games with extracted features
- Stockfish evaluations
- Player ELO ratings

## Prompt Engineering Strategy

### Current Architecture (Implemented)

The system uses a **double-pass architecture**:

1. **Pass 1 - Structured Analysis (JSON)**
   - Input: Game data, Stockfish metadata, SHAP values
   - Output: Validated JSON with structured insights
   - Purpose: Verifiable, hallucination-free analysis

2. **Pass 2 - Narrative Report**
   - Input: Structured JSON + style classification
   - Output: Natural language coaching report
   - Purpose: Human-readable pedagogical feedback

## 🎓 Understanding the System

### Architecture Flow

```
Game → Analysis → SHAP → Competitive Context
                          ↓
                    FACTS_PACK (filtered data)
                          ↓
                    Structured JSON (deterministic)
                          ↓
                    Validation
                          ↓
                    Narrative Prompt (pedagogical)
                          ↓
                    LLM Generation
                          ↓
                    Pedagogical Report
```

### Why Double-Pass Architecture?

**Pass 1 (Structured JSON):**
- Deterministic fact extraction
- No room for hallucination
- Verifiable data points

**Pass 2 (Narrative):**
- LLM explains facts (doesn't invent them)
- Pedagogical tone
- ELO-adaptive language

### Anti-Hallucination Strategy

1. **FACTS_PACK:** Only verified data from engine analysis
2. **Filtering:** Remove meaningless swings (delta < 50 cp)
3. **Strict Rules:** Explicit prohibitions in prompt
4. **Validation:** JSON schema enforcement
5. **Calibration:** Material loss only if verified >2 pawn advantage

## Módulos del proyecto involucrados

- `src/modules/extractor.py` — Extracción básica de features desde FEN
- `src/modules/features_generator.py` — Generación completa de 16 features

---

## 📄 ¿Qué es PGN?

**PGN (Portable Game Notation)** es el formato estándar para registrar partidas de ajedrez. Un archivo PGN contiene:

### Sección de Headers (metadatos)
```
[Event "Torneo de ejemplo"]
[Site "Chess.com"]
[Date "2024.01.15"]
[White "Magnus Carlsen"]
[Black "Hikaru Nakamura"]
[Result "1-0"]
[WhiteElo "2830"]
[BlackElo "2768"]
[ECO "B90"]
[Opening "Sicilian Defense: Najdorf Variation"]
```

### Sección de Movimientos
```
1. e4 c5 2. Nf3 d6 3. d4 cxd4 4. Nxd4 Nf6 5. Nc3 a6 1-0
```

### Headers clave para el análisis:
| Header | Uso en ChessInsightAI |
|:-------|:----------------------|
| `White` / `Black` | Identificar jugadores |
| `WhiteElo` / `BlackElo` | Normalizar predicciones por nivel |
| `ECO` | Clasificar apertura (Encyclopaedia of Chess Openings) |
| `Result` | Ground truth para modelos de predicción |
| `TimeControl` | Distinguir blitz/rapid/classical |

---

## 🔤 Notación FEN (Forsyth–Edwards Notation)

FEN codifica el estado completo del tablero en una sola línea de texto. Es el formato que usa ChessInsightAI internamente para representar posiciones.

### Structure del FEN
```
rnbqkbnr/pppppppp/8/8/4P3/8/PPPP1PPP/RNBQKBNR b KQkq e3 0 1
│                                                │ │    │  │ │
│                                                │ │    │  │ └─ Número de jugada
│                                                │ │    │  └─── Half-move clock
│                                                │ │    └────── En passant target
│                                                │ └─────────── Derechos de enroque
│                                                └───────────── Turno (b=negras)
└────────────────────────────────────────────────────────────── Posición del tablero
```

### Piezas en FEN
| Letra | Pieza | Color |
|:------|:------|:------|
| `K/k` | Rey | Blancas/Negras |
| `Q/q` | Dama | Blancas/Negras |
| `R/r` | Torre | Blancas/Negras |
| `B/b` | Alfil | Blancas/Negras |
| `N/n` | Caballo | Blancas/Negras |
| `P/p` | Peón | Blancas/Negras |

Los números indican casillas vacías consecutivas. `/` separa filas (rank 8 a 1).

---

## ⚙️ Las 16 Features del Modelo

ChessInsightAI extrae **16 características** de cada posición del tablero. Estas features se calculan en `src/modules/features_generator.py` y son la entrada del modelo de ML.

### Features Posicionales (extraíbles sin motor)

| # | Feature | Tipo | Cómo se calcula |
|:--|:--------|:-----|:----------------|
| 1 | `material_balance` | int | Suma ponderada: P=1, N=3, B=3, R=5, Q=9 (blancas - negras) |
| 2 | `material_total` | int | Total de material en el tablero |
| 3 | `num_pieces` | int | Cantidad total de piezas (sin contar reyes) |
| 4 | `branching_factor` | int | `board.legal_moves.count()` |
| 5 | `self_mobility` | int | Movimientos legales del jugador activo |
| 6 | `opponent_mobility` | int | Movimientos legales del oponente |
| 7 | `move_number` | int | Número de jugada en la partida |
| 8 | `player_color` | int | 1 = blancas, 0 = negras |
| 9 | `has_castling_rights` | bool | Si algún bando puede enrocar |
| 10 | `is_repetition` | bool | Si la posición se ha repetido |
| 11 | `is_low_mobility` | bool | Si movilidad < umbral (posición cerrada) |
| 12 | `is_center_controlled` | bool | Control de casillas d4, d5, e4, e5 |
| 13 | `is_pawn_endgame` | bool | Solo peones y reyes en el tablero |
| 14 | `threatens_mate` | bool | Si hay amenaza de mate en 1 |
| 15 | `is_forced_move` | bool | Si solo hay 1-2 movimientos legales |

### Feature del Motor (requiere Stockfish)

| # | Feature | Tipo | Cómo se calcula |
|:--|:--------|:-----|:----------------|
| 16 | `score_diff` | float | `abs(eval_after - eval_before)` en centipawns |

> **score_diff** es la feature más importante del modelo — mide cuánto "empeoró" la evaluación con la jugada realizada vs. la jugada óptima.

---

## 🔧 Procesamiento de Partidas Completas

En el proyecto real, el procesamiento se realiza en lote: se lee un archivo PGN con múltiples partidas y se generan DataFrames con todas las features. El flujo es:

```python
# Pseudocódigo del pipeline real
for game in pgn_file:
    board = game.board()
    for move in game.mainline_moves():
        fen_before = board.fen()
        eval_before = stockfish.evaluate(fen_before)
        board.push(move)
        eval_after = stockfish.evaluate(board.fen())
        
        features = extract_features(board, move_number)
        features['score_diff'] = abs(eval_after - eval_before)
        features['classification'] = classify(features['score_diff'])
        
        dataset.append(features)
```

### Clasificación según score_diff

| Clasificación | score_diff (centipawns) | Categoría |
|:--------------|:------------------------|:----------|
| **Good** | 0 – 30 | Jugada correcta |
| **Inaccuracy** | 30 – 80 | Imprecisión menor |
| **Mistake** | 80 – 200 | Error significativo |
| **Blunder** | > 200 | Error grave |

---

## 📊 Integración con Stockfish

ChessInsightAI integra Stockfish para proporcionar:
- **Evaluación cuantitativa** de cada posición (en centipawns)
- **Mejor movimiento** según el motor (para comparar con el movimiento jugado)
- **Profundidad de análisis** ajustable según disponibilidad de recursos

### Configuration del Motor
```python
from chess import ENGINE

engine = chess.engine.SimpleEngine.popen_uci("./stockfish")
info = engine.analyse(board, chess.engine.Limit(depth=20))
eval_score = info['score'].relative.cp  # Centipawns
best_move = info['pv'][0]  # Principal Variation
```

La feature `score_diff` se calcula como:
```
score_diff = |eval_after_actual_move - eval_after_best_move|
```

Esto mide qué tan diferente es la evaluación si el jugador hubiera jugado lo óptimo en lugar de lo que jugó.
